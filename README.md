## About The Project

Reverie is a cross-platform emotional journaling system — a web application, a mobile app, and a REST API that share a single MongoDB backend. Users write journal entries tagged with a mood state, receive weekly AI-generated reflections powered by Google Gemini, track their emotional patterns on a calendar and analytics dashboard, and seal "memory capsules": time-locked messages that reveal their content on a user-chosen future date.

The project was an exercise in building something production-inspired across three surfaces simultaneously — enforcing consistent authentication, data contracts, and UX patterns whether the user is on a phone or a laptop. It explores what a thoughtfully engineered consumer app looks like when you care about the details: tiered rate limiting, token rotation, graceful AI fallback, offline queuing, biometric lock, and a design system with four mood-driven color atmospheres.

## Why I Built This

Most journaling apps treat writing as a one-way dump. I wanted to explore what it looks like when the app reads back — synthesizing a week of entries into a reflection that actually sounds like it was written for you, not generated by a robot.

Beyond the product idea, the engineering motivation was deliberate: I wanted to build a system that spans three surfaces (API, web, mobile) with shared auth, consistent data contracts, and UI that feels cohesive despite running in a browser and a native app simultaneously. The architectural decisions were intentional learning targets:

- JWT access/refresh token rotation with hashed refresh token storage (not raw tokens in the DB)
- Zod-validated environment config that fails fast on startup rather than at runtime
- Tiered rate limiting per route category (auth, reset, AI generation, general)
- Graceful AI degradation — the system writes a structured fallback reflection if Gemini is unavailable rather than surfacing an error
- Offline-first mobile entry creation with an AsyncStorage queue that syncs on reconnect

## Features

### Core Features
- Rich text journal entries with mood tagging (calm, reflective, hopeful, overwhelmed)
- Weekly AI reflections generated by Gemini 1.5 Flash against the week's entries
- Memory Capsules — time-locked journal entries that remain sealed until a chosen date
- Mood calendar — every entry plotted on a monthly calendar with mood-colored dots
- Mood analytics — distribution charts, streak tracking, word count stats

### Engineering Features
- Dual-surface authentication — HttpOnly cookie (web) and SecureStore token (mobile) from the same auth service
- JWT refresh token rotation with hashed token storage and per-device tracking
- Cron-based weekly reflection generation for all users (Sundays at 9 PM UTC)
- Journal reminder emails — personalized cadence (daily / every 3 days / weekly)
- Offline entry queuing on mobile with expo-network polling and AsyncStorage

### Developer Experience
- Zod schema validation on all environment variables — process exits with clear error messages if misconfigured
- Consistent API response shape (`{ success, data, message, pagination }`) across all endpoints
- Typed request/response throughout — no `any` on API boundaries
- Graceful shutdown with SIGTERM/SIGINT handlers
- Morgan request logging in development, combined format in production

## Architecture

The system follows a conventional three-tier architecture. The backend is a stateless Express API; both the web and mobile clients consume it over HTTP. Authentication state is maintained via short-lived access tokens (15 minutes) and long-lived refresh tokens (30 days), with different storage mechanisms per surface.

```
┌─────────────────────────────────────────────────────────┐
│                     Client Layer                        │
│                                                         │
│  ┌──────────────────┐      ┌──────────────────────┐    │
│  │   Next.js Web    │      │   Expo Mobile App    │    │
│  │  (browser)       │      │  (iOS / Android)     │    │
│  │                  │      │                      │    │
│  │  Auth: HttpOnly  │      │  Auth: SecureStore   │    │
│  │  cookie          │      │  (encrypted)         │    │
│  └────────┬─────────┘      └──────────┬───────────┘    │
└───────────┼──────────────────────────┼────────────────┘
            │  REST + JSON              │
            │  Bearer token             │
            ▼                          ▼
┌─────────────────────────────────────────────────────────┐
│                   Express API (v1)                      │
│                                                         │
│  /auth  /journal  /reflection  /capsules                │
│  /search  /upload  /user                                │
│                                                         │

│           → JWT Auth → Zod Validation                   │

┌─────────────────────────────────────────────────────────┐
│                   MongoDB Atlas                         │
│                                                         │
│  Users  RefreshTokens  JournalEntries                   │
```

### Request Lifecycle (Journal Create)

1. Client sends `POST /api/v1/journal` with `Authorization: Bearer <access_token>`
2. `apiLimiter` checks rate limit (100 req / 15 min window)
3. `authenticate` middleware verifies JWT, confirms user exists in DB, attaches `req.user`
4. `JournalService.createEntry()` creates the document with compound indexes for efficient future queries
5. `sendSuccess(res, { entry }, 'Entry created', 201)` returns the consistent response envelope

### AI Reflection Pipeline

1. Cron fires every Sunday at 9 PM UTC
2. For each user, `ReflectionService.generateReflection()` fetches the week's entries
3. Entry content is stripped of HTML, truncated to 1,500 chars per entry, and injected into a prompt with the emotionally-intelligent system prompt
4. Gemini 1.5 Flash generates the reflection
5. On Gemini failure (quota, auth error, model error), a structured fallback reflection is written instead — the user always gets content
6. Result is stored with mood distribution stats; unique constraint `(userId, weekStart)` prevents duplicates

### Backend

```bash
cd backend
npm install
cp .env.example .env
# Fill in MONGODB_URI, JWT secrets, CLOUDINARY_*, GEMINI_API_KEY
npm run dev
```

### Web

```bash
cd web
npm install
cp .env.local.example .env.local
# Set NEXT_PUBLIC_API_URL=http://localhost:5000/api/v1
npm run dev
```

### Mobile

```bash
cd mobile
npm install
# In lib/api.ts, update BASE_URL to your machine's local IP
# (e.g. http://192.168.1.x:5000/api/v1 — localhost won't work on device)
npx expo start
```

> The mobile app uses `http://localhost:5000` as the default base URL, which works in the iOS Simulator but not on a physical device. Update `BASE_URL` in `mobile/lib/api.ts` to your local network IP when testing on hardware.

## Usage

### Primary Workflows

**Journaling:** Navigate to Write → select a mood → enter title and content in the rich text editor → optionally attach images → Save. Entries are indexed immediately for search.

**AI Reflection:** Navigate to Reflection → Generate reflection. The system reads all entries from the current week, sends them to Gemini with an emotionally-intelligent system prompt, and returns a warm, personal summary. Rate-limited to 5 generations per hour. If Gemini is unavailable, a structured guided reflection is returned instead.

**Memory Capsules:** Navigate to Capsules → New Capsule → write your message → select an unlock date (must be future) → Seal. The capsule's content is masked in all API responses until the unlock date passes. When it unlocks, the `openedAt` timestamp is set on first read.

**Mood Calendar:** Navigate to Calendar. Each day you wrote is dotted with your mood color. Click any day to see that day's entries. The legend shows all four mood colors.

## Known bottlenecks
- The weekly reflection cron calls Gemini serially per user wrapped in `Promise.allSettled`. For large user counts this would be very slow. A queue (BullMQ) would be the right fix.
- Full-text search is implemented as a MongoDB `find()` with optional filters — there's no text index, so it does not search inside `content` or `plainTextContent`. A proper search implementation would use a MongoDB Atlas text index or a dedicated search service.
- No caching layer (Redis) — repeated stats queries hit MongoDB directly.

---

## Security Considerations

**What's implemented:**
- Passwords hashed with bcrypt at cost factor 12
- Raw refresh tokens never stored — only their SHA-256 hashes
- Password reset tokens hashed before storage, expire in 10 minutes
- JWT access tokens expire in 15 minutes; refresh tokens in 30 days with rotation on each use
- Device info and IP address stored per refresh token — enables per-session revocation
- CORS restricted to an explicit origin allowlist
- Helmet.js sets `X-Frame-Options`, `X-Content-Type-Options`, `Strict-Transport-Security`, and others
- Email enumeration prevented on forgot-password — always returns success

## Tradeoffs & Limitations

| Decision | Tradeoff |
|----------|----------|
| Single Express app (no microservices) | Simple to develop and deploy; would not scale individual hot paths independently |
| MongoDB document model | Fast to develop with; joins (aggregations) become verbose for complex analytics |
| Gemini 1.5 Flash | Good quality, free tier; not suitable for production without quota management |
| No Redis / caching | Simpler ops; stats queries hit the database on every request |
| Expo managed workflow | Fast iteration; limited access to native modules |
| Inline capsule service | Fast to write; breaks the controller/service pattern used elsewhere |
| `Promise.allSettled` for batch cron | Resilient to individual failures; but no retry logic or dead-letter handling |

---

## Known Issues

- The weekly reflection cron only runs correctly if the server process is alive on Sunday nights — no persistence or retry if the server is down during the window
- `forceRegenerate` on reflections deletes the existing one before generating the new one — if Gemini fails, the user has no reflection for that week
- Password strength validation exists on signup but not on password reset

---

## Technical Debt

- **Mobile API base URL:** Hardcoded in `mobile/lib/api.ts` rather than driven by an environment variable
- **Search endpoint:** `misc.routes.ts` implements search as a basic filter — no full-text capability, no relevance ranking
- **Reflection prompt version:** `promptUsed` is stored in the DB but there's no versioning scheme — prompt changes would silently affect historical comparisons

---

## Challenges Faced

**Cross-surface authentication consistency:** Making the same JWT auth service work for both a browser (HttpOnly cookie) and a React Native app (SecureStore) from a single Express middleware required careful handling of the `X-Mobile-Client` header to branch cookie vs body behavior.

**AI graceful degradation:** Gemini's error responses are inconsistently structured across quota errors, auth errors, and model errors. The `getGeminiErrorDetails()` helper was written specifically to normalize these into actionable flags (`possibleQuotaIssue`, `possibleInvalidKeyIssue`) so the fallback logic could make informed decisions.

**Offline sync on mobile:** React Native's network state was initially handled via `@react-native-community/netinfo`, which has native linking requirements. Switching to `expo-network` simplified the setup significantly but required rewriting the polling logic to not rely on event subscriptions.

**React Query + Zustand hydration ordering on web:** The Next.js app briefly rendered the authenticated layout for unauthenticated users during client-side hydration before the Zustand store rehydrated from localStorage. The fix was checking `hydrated` state before rendering and returning a loading shell.

## What I Learned

- **Environment validation is not optional.** Zod schema validation on startup saved hours of debugging — misconfigured variables surface immediately with clear messages rather than mysterious runtime errors.
- **Rate limiting needs to be tiered.** A single global limiter doesn't reflect how different endpoints behave. Auth endpoints need tight limits with failure-only counting; AI generation needs hourly budget limits; general API reads need loose limits to not frustrate real usage.
- **The fallback is as important as the happy path.** The AI reflection system is more robust because the fallback produces something meaningful rather than an error. This is a product quality decision, not just an engineering one.
- **Building across three surfaces exposes hidden assumptions.** Assumptions about how tokens are stored, how dates are formatted, and how images are handled all surfaced as bugs when the same feature was implemented on the third surface. Shared contracts (the API response shape, the mood type enum) earn their keep immediately.
- **MongoDB virtual fields are powerful but subtle.** The `isLocked` virtual on `MemoryCapsule` only works if you call `.lean({ virtuals: true })` on list queries — a nuance that cost a few hours.

## Future Scope

- **Full E2E encryption:** Complete the client-side crypto — generate RSA key pairs on signup, encrypt the AES session key per entry with the user's public key, decrypt on read. The data model is already designed for this.
- **Semantic search:** Index entry content in MongoDB Atlas Vector Search or a dedicated vector store, enabling "find entries where I felt anxious about work" queries.
- **Streak rewards:** Gamify consistent journaling — visual streak indicators, milestones, export as a "journaling year in review."
- **Reflection history timeline:** A scrollable visual timeline of weekly reflections with mood trend overlays.
- **Multiple capsule recipients:** Let users send capsules to other people by email, revealed on the unlock date.
- **AI conversation mode:** A follow-up conversational layer on top of a generated reflection — ask "what did you mean by..." and continue the dialogue.
- **Apple Health / Google Fit integration:** Correlate mood data with sleep, exercise, and activity patterns.

---

## Repository Philosophy

This project is production-inspired rather than production-ready. The goal was to make deliberate engineering decisions — not to move fast, but to understand the tradeoffs of each choice. Every pattern used here (tiered rate limiting, token rotation, graceful AI fallback, typed validation on environment config) was a conscious learning target, not incidental complexity.

The codebase is honest about what it doesn't do: there are no tests, no CI/CD pipeline, no monitoring, and one encryption system that was designed but not implemented. These are known gaps, documented plainly.
